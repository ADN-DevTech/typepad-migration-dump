<p>Inspired by <a href="http://through-the-interface.typepad.com/through_the_interface/2012/03/attended-the-windows-8-techconference.html" target="_blank">the Windows 8 conference I attended on Monday</a>, I’ve decided to build my first Metro-style application which scrolls through AutoCAD’s Most-Recently-Used (MRU) drawing list.</p>
<p>I now have a barebones installation of the Windows 8 Consumer Preview + VS11 Beta inside a Parallels VM on my Mac, but rather than installing AutoCAD into that, I decided to write a simple exporter on my existing Windows 7 machine (which now has AutoCAD 2013 installed) to generate the data for my Metro-style application.</p>
<p>Let’s start by looking at what file-oriented MRU data AutoCAD stores, and where.</p>
<p>At the end of each session, AutoCAD stores MRU information about file accesses in the Registry:</p>
<p><a href="http://through-the-interface.typepad.com/.a/6a00d83452464869e2016302dc3464970d-pi" target="_blank"><img alt="MRU information in the Registry" border="0" height="228" src="/assets/image_485684.jpg" style="background-image: none; margin: 20px auto; padding-left: 0px; padding-right: 0px; display: block; float: none; padding-top: 0px; border-width: 0px;" title="MRU information in the Registry" width="394" /></a></p>
<p>For the version of AutoCAD I now have installed it’s stored under this key:</p>
<p>HKEY_CURRENT_USER\Software\Autodesk\AutoCAD\R19.0\ACAD-B001:409\Recent File List</p>
<p>In there you’ll find various information:</p>
<ul>
<li>File1 – File35: the paths to the 35 most recently accessed files (which includes DWGs and sheet sets) </li>
<li>Class1 – Class35: the classification of these files (which so far always seems to be AcApImpRecentDocument) </li>
<li>FilePinned1 – FilePinned35: whether these files have been pinned by the user </li>
<li>FileTime1 – FileTime35: the <a href="http://en.wikipedia.org/wiki/Unix_time" target="_blank">UNIX time</a> stamp indicating when these files were last accessed </li>
</ul>
<p>(This information is based on my observations, and – as is always the case when dealing with the Registry directly rather than a public API – is subject to change.)</p>
<p>Getting the information from this Registry location is pretty straightforward, especially if working from within AutoCAD, as we can use HostApplicationServices.Current.RegistryProductRootKey to provide access to the base Registry location (this has changed to HostApplicationServices.Current.UserRegistryProductRootKey in AutoCAD 2013, where we’ve also exposed a comparable MachineRegistryProductRootKey property from HostApplicationServices).</p>
<p>We need some code that will go through this list and store some information to the file system:</p>
<ul>
<li>A single XML file containing the name of each drawing and information about when it was last accessed </li>
<li>A thumbnail PNG of each drawing (which also needs to be referred to from the XML) </li>
</ul>
<p>A couple of approaches stood out when considering options for doing this “export”:</p>
<ol>
<li>Aside from collecting the file metadata and storing it to XML, generate a script to OPEN, PNGOUT and CLOSE each drawing and then fire up the Core Console to execute it in the background </li>
<li>Create a CRX application to be NETLOADed directly into the Core Console, which can side-load each drawing into a Database to access its properties (including the thumbnail image)</li>
</ol>
<p>I ended up opting for the second approach: while the first is valid for many types of operation, it seemed overkill to manually generate a PNG for each drawing when I could access it from the DWG directly. And I also wasn’t sure whether the act of opening the drawing in the Core Console would modify the MRU or not (I suspect not, as that’s really a UI-related function, but it didn’t seem to be worth finding out).</p>
<p>And besides, I really wanted to see what it was like to implement a CRX and debug using the Core Console rather than full AutoCAD.</p>
<p>A quick refresher… a CRX is more than a DBX but less than an ARX, in that it can be loaded by applications that host <em>AcCore.dll</em> (which includes AutoCAD and the Core Console but not applications using RealDWG for file access). The .crx extension is used for native files – just as .arx and .dbx is – so our .NET class library will simply be a .dll that gets NETLOADed into AutoCAD 2013 or its Core Console.</p>
<p>Creating a Core-capable DLL was easy: I just removed my project reference to <em>AcMgd.dll</em> and added one to <em>AcCoreMgd.dll</em> (making sure the Project’s Reference Paths pointed to AutoCAD 2013 or its ObjectARX SDK, rather than AutoCAD 2012).</p>

<p>You’ll notice the code is very similar to what would have worked in a full AutoCAD application – the main difference was the adjusted namespace to make use of the Application object (Autodesk.AutoCAD.ApplicationServices.<strong>Core</strong>).</p>
<p>To debug the application, I pointed my project settings to the <em>accoreconsole.exe</em> executable (in the same location as <em>acad.exe</em>) and that was it. I could have used the Autoloader or demand-loading to automatically load the DLL, but chose simply to use the NETLOAD command, instead (to which I have an alias defined in my <em>acad.pgp</em> file, so I can use NL rather than typing NETLOAD each time – I was very happy to see the Core Console respect these aliases :-).</p>
<p>Debugging itself was great: it was very quick to launch the app and debug into code (it would have been even nicer if I’d taken the time to set up automatic loading).</p>
