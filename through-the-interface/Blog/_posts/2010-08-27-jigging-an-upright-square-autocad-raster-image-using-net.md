<p>Following on from these <a href="http://through-the-interface.typepad.com/through_the_interface/2010/08/attaching-a-qr-code-raster-image-using-autocad-net.html" target="_blank">two</a> <a href="http://through-the-interface.typepad.com/through_the_interface/2010/08/attaching-an-upright-raster-image-at-a-user-specified-location-in-an-autocad-drawing-using-net.html" target="_blank">posts</a>, we’re now going to implement a jig to create our <a href="http://en.wikipedia.org/wiki/QR_Code" target="_blank">QR Code</a> raster images inside AutoCAD.</p>  <p>Today’s approach isn’t radically different from the previous one prompting selection of corners, but we do get to see the actual (square) boundary of the raster object as it gets defined. It would have been even better if the raster contents were displayed during the jig, but from what I can tell this has been disabled deliberately, no doubt for performance reasons (you see the same effect – only having the boundary visible – when you move or edit a raster via its grips, for instance). I had hoped that, just like for <a href="http://through-the-interface.typepad.com/through_the_interface/2009/03/jigging-an-autocad-block-with-attributes-using-net.html" target="_blank">BlockReferences</a> and <a href="http://through-the-interface.typepad.com/through_the_interface/2009/03/jigging-an-autocad-solid-using-ironpython-and-net.html" target="_blank">Solid3d objects</a>, adding the raster image to the drawing before jigging it – and so passing the RasterImage to the jig, rather than the ObjectId of the RasterImageDef – would cause the contents to be displayed in full, but unfortunately it didn’t change anything.
