<p>Another really interesting, developer-oriented feature in AutoCAD 2013 is something we’ve been calling “Dynamic .NET”. I don’t know whether that’s official branding, or not – I suspect not – but it’s the moniker we’ve been using to describe this capability internally and to ADN members.</p>
<p>The capability is based on an addition to .NET in version 4.0: to complement (or perhaps just as part of) the integration of the <a href="http://en.wikipedia.org/wiki/Dynamic_Language_Runtime" target="_blank">Dynamic Language Runtime</a> into the core .NET Framework, various interfaces – including IDynamicMetaObjectProvider – were provided to developers to let their objects participate in “dynamic” operations.</p>
<p>Providers of APIs therefore now have the option to make certain classes dynamic – for instance, having them <a href="http://msdn.microsoft.com/en-us/library/windows/desktop/ff800651.aspx" target="_blank">implement the IDynamicMetaObjectProvider interface</a> – which means they can effectively be scripted, without binding properties and methods at compile-time. This implements a concept known as <a href="http://en.wikipedia.org/wiki/Duck_typing" target="_blank">duck typing</a>, and is at the core of many scripting language implementations. You may also hear it referred to as late binding: take a look at <a href="http://blogs.msdn.com/b/ericlippert/archive/2012/02/06/what-is-quot-binding-quot-and-what-makes-it-late.aspx" target="_blank">this excellent post by Eric Lippert</a> if you’d like to get a better understanding of what that term means.</p>
<p>So what have we actually done with AutoCAD 2013? We have made Autodesk.AutoCAD.DatabaseServices.ObjectId support dynamic operations: you can declare an ObjectId using the dynamic keyword in C#, and you can then choose to access any of the properties or methods exposed by the object possessing that ID directly from the ObjectId itself. The ObjectId then takes care of the open &amp; close operations implicitly, so there’s no need to mess around with transactions, etc.</p>
<p>If you’re interested in more background to this, I suggest taking a look at <a href="http://au.autodesk.com/?nd=class&amp;session_id=5164" target="_blank">Albert Szilvasy’s class from AU 2009</a>, where he gave us a hint of what was to come (although it was far from being a certainty, at that time – we have since found it to be a popular option via our annual API wishlist surveys).</p>

<p>Something that needs to be stressed: as the properties and methods are effectively being determined and then accessed/called at run-time, there is no <a href="http://en.wikipedia.org/wiki/IntelliSense" target="_blank">IntelliSense</a> available when coding dynamically (something Python and Ruby developers are used to, but we statically-typed .NET developers are probably less so). It would not be impossible to implement, but it would be very difficult: aside from types being inferred at the time code is being written (I’ll use the term develop-time, although I haven’t seen it used before), there would need to be some work done to effectively populate the list of properties &amp; methods, which would typically mean opening the object and reflecting over its protocol… not at all straightforward to implement, as far as I can tell, especially when you don’t actually have AutoCAD running to help with accessing the objects.</p>
<p>A lack of IntelliSense does mean this style of coding it likely to be less accessible to novice coders, unless a more integrated <a href="http://en.wikipedia.org/wiki/REPL" target="_blank">REPL</a> environment becomes available (as this is something that tends to balance out the lack of other develop-time tooling when scripting – just as people have found with the ability to execute fragments of AutoLISP at AutoCAD’s command-line, over the years).</p>
<p>I can see a couple of ways a REPL might end up being implemented. One would be to host the DLR and (say) the IronPython scripting engine (just as Albert showed during his above-mentioned AU class, as I did at <a href="http://au.autodesk.com/?nd=class&amp;session_id=5125" target="_blank">my own AU 2009 class</a>, and we’ve seen <a href="http://through-the-interface.typepad.com/through_the_interface/2009/12/command-line-scripting-of-ironpython-code-in-autocad.html" target="_blank">previously on this blog</a>). Another would be to wait for the fruits of the future .NET Framework feature codenamed <a href="http://blogs.msdn.com/b/ericlippert/archive/2010/12/16/hiring-for-roslyn.aspx" target="_blank">Roslyn, which enables Compiler as a Service for .NET&#0160; code,</a> as this seems it would be a likely way to execute C# and VB.NET in a more dynamic way.</p>
<p>But that’s all for the future, and is very much my own speculation, at this stage.</p>
<p>One question that has come up in relation to “Dynamic .NET” in AutoCAD is around performance: as we’re effectively opening and closing an object every time we execute a method (and we may actually open and close an object twice – once for read, once for write – when we read one of its properties, modify it and store it back), isn’t there a performance impact? I admit I haven’t performed extensive benchmarking of this, and there is likely to be some overhead if dealing with lots of properties and methods. But in terms of getting something working quickly – which you can then tune for performance later, as needed – this is likely to be a valuable tool for developers, irrespective of the performance impact (which I’d hope to be modest).</p>
