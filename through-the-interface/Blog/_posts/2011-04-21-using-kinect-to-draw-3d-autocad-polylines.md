<p>As promised in <a href="http://through-the-interface.typepad.com/through_the_interface/2011/04/previewing-and-capturing-kinect-sensor-data-directly-inside-autocad.html" target="_blank">this previous post</a>, I’ve been playing around with understanding Kinect gestures inside AutoCAD. My first step – after upgrading the <a href="http://openni.org" target="_blank">OpenNI</a> modules and drivers to more recent versions, which always seems time-consuming, for some reason – was to work out how to get gesture information via <a href="http://nkinect.codeplex.com/" target="_blank">nKinect</a>. It turned out to be very straightforward – as it’s based on OpenNI and <a href="http://www.primesense.com/?p=515" target="_blank">NITE</a>, the user-tracking and gesture detection come pretty much for free.</p>
<p>A few things I had to work out:</p>
<ul>
<li>We needed a “skeleton callback”, which is called when body movements are detected      
<ul>
<li>Because this is fired on a non-UI thread, we need to be careful <a href="http://through-the-interface.typepad.com/through_the_interface/2008/06/cleaning-up-aft.html" target="_blank">not to use it to create DBObject-derived objects</a> </li>
<li>We get around this by create LineSegment3d objects which we then convert to Lines for transient display later on</li>
</ul>
</li>
<li>It takes time for NITE to detect and start tracking a user – until this happens the above callback doesn’t get called </li>
</ul>
<p>Once I’d understood the mechanism, it was reasonably straightforward to track the position of the user’s right hand and draw – both transiently and more permanently – a set of line segments tracking the hand’s movement. The Transient Graphics sub-system ended up proving a better choice for displaying our temporary line segments than the jig, as the line segments are actually pretty static with new segments just building on the ones drawn previously.</p>
