<p>Before the break, I had <a href="http://through-the-interface.typepad.com/through_the_interface/2013/07/minor-update-to-the-dgn-purge-command.html?cid=6a00d83452464869e2019b028845bb970c#comment-6a00d83452464869e2019b028845bb970c" target="_blank">a characteristically insightful comment</a> from Gilles Chanteau on a post regarding the DGNPURGE implementation that was released as <a href="http://through-the-interface.typepad.com/through_the_interface/2013/07/autocad-dgn-hotfix-now-available.html" target="_blank">a Hotfix for AutoCAD 2013 and 2014</a>. As usual with these things, sometimes you don’t see the wood for the trees once you’ve started down a particular path. Here’s the comment in question:</p>  <blockquote>   <p><em>Just a thought, you use: for (long i = 1; i &lt; handseedTotal; i++) ...        <br />to scan the whole database, it's a nice way i use too searching for proxies for example. But here you're looking only for entities to check their owners.         <br />Won't it be more efficient to only iterate all BlockTableRecords?</em></p> </blockquote>  <p>In this case, Gilles cast fresh eyes on the solution and indeed found a better way to implement it (I ended up changing very few lines of code but it certainly made the implementation cleaner and much more efficient).</p>  <p>Today’s post shares this implementation – which at this stage probably won’t make it into an update to the DGN Hotfix, but people can certainly build the code from this post into an updated DLL, should they see the benefit – but also discusses a problem I spent some time troubleshooting earlier today.</p>  <p>The issue had been raised by Brock Priebe just before the break in <a href="http://through-the-interface.typepad.com/through_the_interface/2013/07/minor-update-to-the-dgn-purge-command.html#comment-6a00d83452464869e2019b033220f0970d" target="_blank">a blog comment</a> on the same post, but it wasn’t something I could reproduce from my side: sometimes the DGN linestyle-related objects just end up turning into zombies. (Please don’t worry: the walking dead aren’t attacking along with <a href="http://through-the-interface.typepad.com/through_the_interface/2014/01/attack-of-the-robots.html" target="_blank">the killer robots</a>… for those unaware of the history, proxy objects were originally called zombies, back before the terminology was sanitized by the marketing department. ;-)</p>  <p>Anyway, the point is that the DGNPURGE tool simply isn’t designed to delete the proxy objects that get created when the DGN linestyle strokes aren’t resurrected on drawing open. For each inaccessible object, it reports:</p>  <p><font size="1" face="Courier New">Unable to erase stroke (AcDbZombieObject): eNotAllowedForThisProxy.</font></p>  <p>In passing, just before the break, <a href="http://through-the-interface.typepad.com/through_the_interface/2013/07/minor-update-to-the-dgn-purge-command.html#comment-6a00d83452464869e2019b03343df7970c" target="_blank">I pointed Brock to the Zombie Killer app</a> that Gilles has <a href="http://apps.exchange.autodesk.com/acd/en/Detail/Index?id=appstore.exchange.autodesk.com%3azombiekiller_windows32and64%3aen&amp;autostart=true" target="_blank">published to Autodesk Exchange</a> (nice bit of circularity, there :-), which apparently did help get rid of the DGN-related proxy objects in this situation.</p>  <p>But the question came up again, this afternoon, both <a href="http://forums.autodesk.com/t5/AutoCAD-Civil-3D-General/DGNPURGE-doesn-t-work-on-all-computers/td-p/4737589" target="_blank">on the discussion group</a> and in an email from Jason Olesky. Jason had found that this problem wasn’t drawing-specific, it was actually machine-specific. I’d seen it reported once or twice before, but had assumed it was due to drawing corruption rather than a problem with certain product installations.</p>  <p>We discussed back and forth by email and ended up realising that certain systems – some of which have been newly installed, which I find worrying – simply don’t have the Registry entries to demand-load the <em>AcDgnLS.dbx</em> module when DGN linestyle information is found in a DWG. Jason found that copying the information across from a functioning system allowed this to work properly.</p>  <p>Here’s the Registry export from my system, in case it helps someone:</p>  <p><font size="1" face="Courier New">Windows Registry Editor Version 5.00</font></p>  <p><font size="1" face="Courier New">[HKEY_LOCAL_MACHINE\SOFTWARE\Autodesk\ObjectDBX\R19.1\Applications\AcDgnLS]      <br />&quot;LOADCTRLS&quot;=dword:00000009       <br />&quot;LOADER&quot;=&quot;AcDgnLS.dbx&quot;       <br />&quot;DESCRIPTION&quot;=&quot;DGN Line Style Component&quot;       <br /></font></p>  <p>I still don’t know why the Registry entries don’t get created for some AutoCAD Civil 3D 2014 installations (I wonder if there isn’t a problem with the second-stage installer, given that this is a key under HKLM rather than HKCU and Jason mentioned that a number of the users of problematic machines in his organisation hadn’t actually run C3D before). In any case, knowing there’s a workaround will hopefully be of help to people who run into this situation.</p>  <p>Incidentally, you will get exactly the same behaviour if you set DEMANDLOAD to 0 (rather than the default value of 3, which allows DBX and ARX modules to be demand-loaded on proxy detection and command invocation, respectively).
