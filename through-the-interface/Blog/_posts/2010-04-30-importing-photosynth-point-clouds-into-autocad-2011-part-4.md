<p>In the previous posts in this series we <a href="http://through-the-interface.typepad.com/through_the_interface/2010/04/importing-photosynth-point-clouds-into-autocad-2011-part-1.html" target="_blank">introduced a command that downloaded and imported point clouds</a> from <a href="http://photosynth.net" target="_blank">Photosynth.net</a>, we introduced <a href="http://through-the-interface.typepad.com/through_the_interface/2010/04/importing-photosynth-point-clouds-into-autocad-2011---part-2.html" target="_blank">a WinForms user interface on top of it</a> and then <a href="http://through-the-interface.typepad.com/through_the_interface/2010/04/importing-photosynth-point-clouds-into-autocad-2011---part-3.html" target="_blank">replaced that UI with one implemented using WPF</a>.</p>
<p>As threatened last time, we’re now going to make some efficiency improvements in the original command implementation.</p>
<p>In our previous implementation we were blindly asking for files, one after the other, and using failure to indicate when we’d reached the end. Which was fine, but it limited us in a few ways: we could not reliably parallelize this otherwise highly parallelizable operation, and we couldn’t report accurate progress back to the user (as we didn’t know when it was all going to end).</p>
<p>Thanks to <a href="http://through-the-interface.typepad.com/through_the_interface/2010/04/importing-photosynth-point-clouds-into-autocad-2011-part-1.html#comment-6a00d83452464869e201347ff26d8a970c" target="_blank">a note from Nate Lawrence</a> and another look at <a href="http://pspcexporter.codeplex.com" target="_blank">Christoph Hausner’s Photosynth Point Cloud Exporter project</a>, I was able to work out how to get this information from <a href="http://photosynth.net/photosynthws/PhotosynthService.asmx" target="_blank">the Photosynth web service</a>. The beauty of getting this information is that we now know exactly what files we need to download and can fire them off as asynchronous tasks.</p>
<p>The best way I know of managing this kind of activity is by integrating F# into your project, and this is – in my opinion – one of the absolutely compelling benefits of the F# language: it’s just so easy to capture the logic of “Asynchronous Workflows”, such as this, and to leave the F# subsystem to execute them as efficiently as it can. And, as we’ll see, for a task where we’re downloading and processing multiple files there are huge performance benefits versus performing this sequentially.</p>
<p>Before we look at the code, a few notes on connecting to the Photosynth web service. F# projects – at least with the <a href="http://www.microsoft.com/downloads/details.aspx?FamilyID=444005FB-E627-4FEB-B51D-13D6A3B4B8ED&amp;amp;displaylang=en&amp;displaylang=en" target="_blank">April 2010 CTP</a> I’m using with VS 2008 – do not have IDE support for adding web service references, so I decided to keep this “discovery” activity in C#.</p>
<p>Adding service references to a C# project is easy – we right-click inside the Solution Explorer and select “Add Service Reference…”, copying/pasting the web service URL (<a href="http://photosynth.net/photosynthws/PhotosynthService.asmx">http://photosynth.net/photosynthws/PhotosynthService.asmx</a>) into the Address bar:</p>

<p>A few comments on the code:</p>
<ul>
<li>When running this code a lot in the debugger, I found we were getting lots of hidden instances of our browser executable (as we’re killing it rather than allowing it to exit naturally, due to a bug in the <a href="http://code.google.com/p/csexwb2/" target="_blank">csExWb2</a> component), so I decided to provide some automatic clean-up functionality on startup of the application </li>
<li>We’ve removed a lot of code dealing with the downloading/processing of point data – this is all now taken care of by the F#-implemented PointCloudProcessor object </li>
<li>This C# code now depends on an additional library called <a href="http://james.newtonking.com/pages/json-net.aspx" target="_blank">Json.NET</a> (as did the exporter from which I borrowed the code) to parse <a href="http://en.wikipedia.org/wiki/JSON" target="_blank">JSON</a>-formatted information we download from the Photosynth web service </li>
<li>We have a situation where we want to perform asynchronous operations on arbitrary threads, but these operations need to report back to the UI thread in order for us to write text to the command-line and to update AutoCAD’s progress meter. For this we have to make sure we have a valid <a href="http://www.codeproject.com/KB/cpp/SyncContextTutorial.aspx" target="_blank">SynchronizationContext</a> set-up, which will be used from our F# code </li>
</ul>

<p>Some comments on this code:</p>
<ul>
<li>A big chunk of this implementation has been copied verbatim from <a href="http://blogs.msdn.com/dsyme/archive/2010/01/10/async-and-parallel-design-patterns-in-f-reporting-progress-with-events-plus-twitter-sample.aspx" target="_blank">Don Syme’s excellent post on reporting progress from asynchronous tasks</a>. The main change to Don’s AsyncWorker implementation is our need to fail should we be unable to get a valid SynchronizationContext for our main thread: if this doesn’t work then we need to fail (gracefully), as we will not be able to report our progress via a new context executing in <a href="http://msdn.microsoft.com/en-us/library/system.threading.threadpool.aspx" target="_blank">the .NET thread pool</a> </li>
<li>Otherwise we have a number of <a href="http://en.wikipedia.org/wiki/Tail_recursion" target="_blank">tail-recursive functions</a> (a topic <a href="http://through-the-interface.typepad.com/through_the_interface/2008/02/more-random-mus.html" target="_blank">I’ve discussed previously</a>), to avoid both iterative code and stack overflows </li>
<li>We also expose an event which is subscribed to in our C# calling code. This means we don’t need any dependency on AutoCAD libraries in the F# project: to update the command-line and progress meter we simply need to fire that event from the UI thread and let the code execute from the project with the appropriate assembly references </li>
<li>We’re using another asynchronous concept to manage our writing to the local text file for our point data: we’re using an <a href="http://through-the-interface.typepad.com/through_the_interface/2009/11/using-erlang-style-message-passing-from-f-to-coordinate-asynchronous-tasks-in-autocad.html" target="_blank">agent</a> to manage this </li>
</ul>
