<p>I wasn’t planning on covering this particular topic today, but then <a href="http://through-the-interface.typepad.com/through_the_interface/2013/06/inserting-a-specific-excel-sheet-as-an-autocad-table-using-net.html#comment-6a00d83452464869e201a3fcb496a5970b" target="_blank">this comment</a> came in overnight and I ended up taking a look into it. Paul has been trying to append a specific cell range to his connection string when creating a data-link for an Excel spreadsheet, adding to the code from <a href="http://through-the-interface.typepad.com/through_the_interface/2013/06/inserting-a-specific-excel-sheet-as-an-autocad-table-using-net.html" target="_blank">this previous post</a>.</p>  <p>I gave it a try myself with <a href="http://through-the-interface.typepad.com/through_the_interface/2013/06/inserting-a-specific-excel-sheet-as-an-autocad-table-using-net.html#comment-6a00d83452464869e201a511647d30970c" target="_blank">a hardcoded cell range</a> and it seemed to work fine, and so went ahead and modified the implementation of the TFS command to ask the user to enter their own cell range. What’s probably most interesting about the code in this post is its use of a <a href="http://en.wikipedia.org/wiki/Regular_expression" target="_blank">regular expression</a> – something I don’t use often enough, in truth – which is a great way of validating that strings conform to a particular pattern.</p>  <p>It’s also possible to apply group labels within a regex to tag groups of characters for later extraction and validation. We use this technique to check that the row numbers are both greater than 0, for instance (“A0:B3” and “A10:B0” are both invalid cell ranges that otherwise meet the pattern we define).</p>  <p>There may well be a better way to do this within a regex – as I’ve said, I don’t use them as often as I should – but hopefully the overall technique of using them will be a helpful reminder for people.</p>  <p>On a more general note, this kind of input validation is extremely important if you’re asking for user input and then using it to build database queries: if you don’t protect against arbitrary strings being entered then your application will be wide open to <a href="http://en.wikipedia.org/wiki/SQL_injection" target="_blank">SQL Injection</a> attacks. This isn’t something that’s as important with desktop applications accessing local databases, perhaps, but as applications move more to the web this becomes absolutely critical (as <a href="http://sitesdoneright.com/blog/2013/11/sql-injection-hacking-on-healthcare-gov" target="_blank">the unfortunate autocomplete snafu on healthcare.gov</a> highlighted a couple of months ago).</p>  <p>Here’s the updated C# code... the TFS command is the only section that’s had significant changes from the previous version, in case:</p>  <div style="font-family: courier new;
