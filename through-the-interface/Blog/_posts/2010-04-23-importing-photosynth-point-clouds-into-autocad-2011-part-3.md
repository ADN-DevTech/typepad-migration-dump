<p>As alluded to in <a href="http://through-the-interface.typepad.com/through_the_interface/2010/04/importing-photosynth-point-clouds-into-autocad-2011---part-2.html" target="_blank">the last post in this series</a> (ignoring <a href="http://through-the-interface.typepad.com/through_the_interface/2010/04/adding-to-autocads-application-menu-and-quick-access-toolbar-using-net.html" target="_blank">a related post that dealt with user interface integration</a>) I wasn’t really happy with some of the tricks I needed in the <a href="http://en.wikipedia.org/wiki/Windows_Forms" target="_blank">WinForms</a> version to try and make a coherent user interface for tracking accessed point clouds in a hosted Photosynth browsing session. This post replaces the WinForms UI with one implemented using <a href="http://en.wikipedia.org/wiki/Windows_Presentation_Foundation" target="_blank">WPF</a>, and in fact might also have been titled “Using data-binding in WPF to track a list of objects with associated thumbnails” or something to that effect. :-)</p>
<p>What I’ve done in the <a href="http://through-the-interface.typepad.com/files/ImportPhotosynth2.zip" target="_blank">new version of the solution used to build this tool</a> is to add a separate WPF application project (called BrowsePhotosynth) while leaving the WinForms project there as a historical reference (that one is still called Browser). The new WPF project builds an executable called “ADNPlugins-BrowsePhotosynth2.exe”, which has made it easy to update the DLL project to work with this new version – all we have to do is add a “2” in a few places. Otherwise the code in the main DLL module remains unchanged (and won’t be listed in this post). Please refer back to the previous post for instructions on getting the application to work: if you have already got the last version working, you should be able to drop the new DLL and EXE into the same folder and they should just work.</p>
<p>Let’s now take a look at the pertinent files from the project for our new-fangled WPF browser. Bear in mind that I’m far from being a WPF expert, myself (I do feel as though I’m making progress, though), so I’ve relied a lot on code I’ve found on the web for various tasks: <a href="http://www.developingfor.net/wpf/accessing-command-line-arguments-in-wpf.html" target="_blank">accessing command-line arguments</a>, <a href="http://www.codeproject.com/KB/cs/BindBetterINotifyProperty.aspx" target="_blank">automatically updating the UI when bound properties change</a>, <a href="http://learnwpf.com/post/2006/05/08/How-Do-I-Add-a-Splitter-to-my-WPF-Window.aspx" target="_blank">adding a splitter</a>, <a href="http://rndnext.blogspot.com/2008/12/set-wpf-listbox-selecteditem-on-hover.html" target="_blank">enabling selection on hover</a>, <a href="http://www.uxpassion.com/2008/09/styling-wpf-listbox-highlight-color/" target="_blank">using a gradient selection style</a> and <a href="http://stackoverflow.com/questions/135518/using-wpf-what-is-the-best-method-to-update-the-background-for-a-custom-button-c" target="_blank">changing a button’s colour when being hovered over</a>. Which means the code may have redundant properties set or use inconsistent approaches to solving the various design problems, but anyway. We’re all learning. :-)</p>

<p>The comments should go some way to explaining the logic behind the design.</p>
<p>The “code-behind” file is actually in many ways less complicated than its predecessor, as we aren’t having to deal with custom draw code (that’s taken care of in the XAML, now).</p>
<p>Here’s a summary of the main enhancements/optimizations:</p>
<ul>
<li>We now use data-binding to display the information we have about our synths 
<ul>
<li>In the previous version we had to maintain various lists (the synth data, the images and the visible list), so this has simplified things greatly</li>
<li>We had a little more plumbing to do to make sure our properties implement INotifyPropertyChanged properly, so that the UI gets updated automatically when the properties change</li>
<li>We also have to use an ObservableCollection&lt;&gt; rather than a List&lt;&gt; for our SynthInfo objects, which allow the notifications to propagate properly. Luckily this doesn’t result in a change in the XML data stored for our synths, which is cool</li>
<li>WPF had issues with dependency properties such as this being updated from other threads, so we no longer use a timer to update our images 
<ul>
<li>Dispatcher.Invoke() is actually a better way for queuing up this task, in any case, and makes life much simpler</li>
</ul>
</li>
</ul>
</li>
<li>Our Window is constructed via App.xaml, and so can no longer take arguments 
<ul>
<li>We now set the URL in our form constructor and get AutoCAD’s HWnd from the app object, where it gets set as a static variable on startup</li>
</ul>
</li>
<li>Our events are now hooked up via XAML, which streamlines our form’s constructor</li>
<li><a href="http://code.google.com/p/csexwb2/issues/detail?id=59" target="_blank">The csExWb control causes an annoying COM exception in a WPF application on shutdown</a>, which is probably a clue to the issue we were seeing with the WinForms application (and originally with AutoCAD, when I had it hosted in a NETLOADed DLL) 
<ul>
<li>Because the exception is more visible from WPF, we don’t ever exit the application, even when the “X” is used by the user</li>
<li>We now just hide the form and let AutoCAD kill the process when the BP command is next called</li>
</ul>
</li>
<li>This version of the application doesn’t have a right-click menu for importing the point clouds 
<ul>
<li>This is should be a trivial thing to add back, in case</li>
</ul>
</li>
<li>In terms of the UI changes, we have a nice little slider that allows the user to change the size of the images in the list, which is quite handy 
<ul>
<li>It allows you to zoom into the image to help decide whether to import it</li>
</ul>
</li>
</ul>
