<p>After a break of a week, I thought it was time to take the <a href="http://en.wikipedia.org/wiki/QR_Code" target="_blank">QR Code</a> application a little further, after our previous versions creating basic QR codes <a href="http://through-the-interface.typepad.com/through_the_interface/2010/08/attaching-a-qr-code-raster-image-using-autocad-net.html" target="_blank">at a fixed location</a>, <a href="http://through-the-interface.typepad.com/through_the_interface/2010/08/attaching-an-upright-raster-image-at-a-user-specified-location-in-an-autocad-drawing-using-net.html" target="_blank">using user-specified corners</a> and <a href="http://through-the-interface.typepad.com/through_the_interface/2010/08/jigging-an-upright-square-autocad-raster-image-using-net.html" target="_blank">using a jig</a>.</p>  <p>The code in this post adds quite a bit of functionality to the application:</p>  <ul>   <li>The ability to encode various types of data      <ul>       <li>Calendar events </li>        <li>Contact information </li>        <li>Email addresses </li>        <li>Geo-locations </li>        <li>Phone numbers </li>        <li>Plain text </li>        <li>URLs </li>     </ul>   </li>    <li>The ability to edit QR Codes      <ul>       <li>The data used to create the QR Code is attached to the raster image as XData and gets used for default values </li>     </ul>   </li>    <li>The ability to query the URL of an existing QR Code      <ul>       <li>i.e. the Google Code URL used to create the raster image </li>     </ul>   </li>    <li>The ability to decode an existing QR Code      <ul>       <li>The Google ZXing Decoder is launched automatically with the URL of the selected QR Code </li>        <li>Handy for checking the QR Code just after creation </li>     </ul>   </li> </ul>  <p>The UI is still command-line based, but it’s altogether possible to bolt on a fancy GUI, should one so wish (and I may well extend that in a future post, we’ll see). In the main part the UI options have been modelled after those available on the <a href="http://zxing.appspot.com/generator/" target="_blank">ZXing Generator</a> (leaving out a few data-types that don’t seem very relevant, such as “SMS” and “WiFi network”).</p>  <p><a href="http://through-the-interface.typepad.com/files/QrCodes-Project.zip" target="_blank">The source project</a> now contains a number of different C# files:</p>  <ul>   <li><em>Commands.cs</em> – our command definitions </li>    <li><em>QrInput.cs</em> – user input functions for our command-line data entry </li>    <li><em>QrEncoder.cs</em> – encoder functions to generate a Google Chart URL for a set of data </li>    <li><em>RbEncoder.cs</em> – encoder functions to create a ResultBuffer from the user-specified data </li>    <li><em>demand-loading.cs</em> – our usual file to auto-generate demand-loading Registry keys </li> </ul>  <p>There’s nothing very remarkable about the code: it did occur to me as I was hard-coding the input paths for the various data-types that it might be interesting to generalise the mechanism to use some kind of “schema” of the various data fields and use a more dynamic approach to query and encode the information. But I decided not to go down that path, for now, at least.</p>  <p>I decided to implement a custom exception type in <em>QrInput.cs</em> to denote user cancellation, mainly because it was proving cumbersome to force the return type to encode that condition in some way (such as a null string or Double.NaN), and I didn’t want to fall back on ref or out parameters. So when the user cancels we accept the overhead of throwing and handling an exception, which certainly makes the code simpler.</p>  <p>There’s still potential for replacing the generation of the QR Codes themselves to be performed locally, rather than via a web service. I think – overall – that the benefits of the current approach do outweigh the effort required to use a local QR Code library, but this is certainly something that could change (feedback welcome!).</p>
