<h4>Jeremy Tammik</h4>

<p>Jeremy is
<a href="http://thebuildingcoder.typepad.com">The Building Coder</a> and
<a href="http://the3dwebcoder.typepad.com">The 3D Web Coder</a>.
He is a prolific author and passionate about cooperation and sharing, working with the Forge Partner Development team on Autodesk APIs and web services, providing developer support, training, conferences, presentations, and blogging on the Revit API and cloud and mobile technologies.</p>

<p>Jeremy joined Autodesk in 1988 as the European technology evangelist driving AutoCAD application development in Europe, the United States, Australia, and Africa. He co-founded ADGE, the AutoCAD Developer Group Europe. He left Autodesk in 1994 to work as an independent HVAC application developer and rejoined the company in 2005.</p>

<p>Jeremy graduated with degrees in mathematics and physics at Marburg University in Germany, worked as a teacher and translator of both computer and human languages and as a C++ programmer on early GUI and multitasking projects. <h4>Getting Started</h4>

<p>Here are some suggestions on getting started with the Revit API:</p>

<ul>
<li><a href="https://thebuildingcoder.typepad.com/blog/2018/06/extensible-storage-and-renaming-a-family-in-a-project.html#2">Before getting started</a></li>
<li><a href="https://thebuildingcoder.typepad.com/blog/about-the-author.html#2">Getting started material</a></li>
<li><a href="https://thebuildingcoder.typepad.com/blog/2017/01/virtues-of-reproduction-research-mep-settings-ontology.html#3">How to research to find a Revit API solution</a></li>
</ul>

<p>First of all,
<a href="http://thebuildingcoder.typepad.com/blog/2018/06/extensible-storage-and-renaming-a-family-in-a-project.html#2">before getting started</a>,
learn the Revit product from an end user point of view.
Understand the optimal workflow and best practices.
If you do not, the API will be hard to understand and not make much sense.</p>

<p>It is also useful to learn the basic of programming in general, and of the language you choose for addressing the Revit .NAT API specifically.</p>

<p>A good understanding of event-driven programming is very helpful, since the Revit API is entirely event driven.</p>

<p>Once you have that under the hood, determine an optimal end user workflow for the task that you intend to address with your add-in.</p>

<p>Any step that cannot be solved through the user interface can almost certainly not be automated either.</p>

<p>After that, it is time to start exploring the programming environment.</p>

<p>The Revit SDK or Software Development Kit includes API documentation and samples.
You can download it from the Autodesk <a href="http://www.autodesk.com/developrevit">Revit Developer Center</a>.</p>

<p>The most important piece of documentation is the Revit API help file <i>RevitAPI.chm</i> listing all classes, properties and methods. That information is also available online at <a href="http://www.revitapidocs.com">www.revitapidocs.com</a>.</p>

<p>Another very important piece of documentation is
the <a href="http://help.autodesk.com/view/RVT/2024/ENU/?guid=Revit_API_Revit_API_Developers_Guide_html">Revit API Developers Guide</a> included in the Revit product help.

<p>The most effective way to get started learning and programming with the Revit API, including installation and use of the SDK, is to work through the step-by-step instructions provided by the <i>DevTV</i> and <i>My First Revit Plugin</i> video tutorials.</p>

<p>Next, you can dive in deeper still by working through the self-documenting
<a href="https://github.com/ADN-DevTech/RevitTrainingMaterial"><i>ADN Revit API Training Labs</i></a>.
They consist of a series of hands-on lab exercises in both C# and VB.NET accompanied by detailed step-by-step training instruction documents.</p>

<p>If you are in a hurry and want to skip everything else, you can just check out
the <a href="http://help.autodesk.com/view/RVT/2024/ENU/?guid=Revit_API_Revit_API_Developers_Guide_Introduction_Getting_Started_Walkthroughs_html">hello world walkthroughs in the developer guide</a>.</p>

<p>The section on <a href="http://help.autodesk.com/view/RVT/2024/ENU/?guid=Revit_API_Revit_API_Developers_Guide_Introduction_Add_In_Integration_html">Add-In Integration</a> presents detailed information on all you need to know about installing an add-in.</p>

<h4>Reproducible Test Case</h4>

<p>The best and clearest way to describe an API problem is normally by providing a non-confidential reproducible test case including a minimal sample macro embedded in a minimal project file to run it in and detailed step-by-step instructions specifying exactly what you are trying to achieve, the behaviour you observe, the difference between the two and how to reproduce the issue to analyse it in the debugger:</p>

<ul>
<li>A <b>short</b> exact description of what you are trying to achieve.</li>
<li>The behaviour you observe versus what you expect, and why this is a problem.</li>
<li>A complete yet minimal Revit sample model to run a test in.</li>
<li>A complete yet minimal macro embedded in the sample model or Visual Studio solution with add-in manifest that can be compiled, loaded, run and debugged with a single click to analyse its behaviour live in the sample model.</li>
<li>Detailed step-by-step instructions for reproducing the issue, e.g. which element to pick, what command to launch etc.</li>
</ul>

<p>The purpose of this is both to enable us to <b>reproduce</b> and <b>analyse</b> the problem and to <b>verify</b> that it really has been resolved.</p>

<p>Another even more beneficial aspect for all involved is that creating a minimal reproducible case like this can help you discover for yourself what you were doing wrong, and the issue becomes moot.</p>

<p style="font-size:small">P.S. Here is an additional clarification on the importance of a minimal reproducible case directly from the development team for a specific case: With the new info it does indeed look as if we need to explore this issue in more depth. However, I doubt it is on the managed side. Unfortunately, even if it does not look like it, we really need a sample to reproduce. Unless the problem occurs every time no matter what geometry is and what happens to the element the face was taken from, it is practically impossible for us to test it (I mean, in a reasonable time). We do not necessarily need your code &ndash; I can write the code myself &ndash; but we need a workflow to reproduce, and the workflow needs to be quite detailed; like &ndash; create this wall, of this type and this position and length, at this level; put this specific window there at this exact location; etc. Let me put it this way: if I receive a case from our QA team and the description does not specify each click exactly to get to the point of failure, no matter how trivial the commands are, I cannot take the case and return it to the reporter. Naturally I always try my best to deduce the actual scenario, but I can only do so much. So, please, if you can give us something very detailed and specific it will benefit both of us. Thank you.</p>

<p style="font-size:small">P.P.S.
If your add-in is too large or complex:
I would love to dive in and try to help you debug this, but I am sorry to say I do not have the time.
You will have to continue exploring it yourself.
All I can suggest is to keep at it.
One approach to debugging a problem like this is:
(i) Simplify it down to something absolutely trivial and stupid that is guaranteed to work &ndash; dumb it down.
(ii) Once that is working, add the required complications one by one until it either works completely or fails.
Once you have determined the exact point of failure, you can narrow that down further and create a minimal reproducible case:
With the <u>minimal</u> reproducible case in hand, I can either take a look myself of pass it on to the Revit development team for further analysis.</p>

<p style="font-size:small">P.P.P.S. In order to simplify installation and debugging for the development team, the preferred method to submit your sample code snippet is to include it as a macro embedded in a minimal RVT document.</p>
