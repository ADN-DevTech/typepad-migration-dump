<p>By <a href="http://adndevblog.typepad.com/autocad/augusto-goncalves.html">Augusto Goncalves</a></p>
<p><strong>What&#39;s coming</strong></p>
<p>Have you noticed that the Autodesk Exchange Store website is showing its UI interface in non-English? From the website perspective, it will automatically select the correct version according to the ‘Language’ settings of the browser, or you can select at the bottom of the page.</p>
<p>Now the interesting point: we’ll soon start accepting applications that are in other languages, including multi-language or localized only (e.g. German only apps).</p>
<p>This post will show how make your application work for multi-language using the <a href="http://adndevblog.typepad.com/autocad/2013/01/autodesk-autoloader-white-paper.html">Autoloader</a> mechanism.</p>
<p><strong>Background</strong></p>
<p>The AutoCAD command registration can work with names in English and non-English, called Global and Local names, respectively. With .NET API we can implement this feature with Resources, as described <a href="http://through-the-interface.typepad.com/through_the_interface/2009/06/registering-autocad-commands-with-localized-names-using-net.html">here</a>, which work for all supported languages on the same assembly, also can be used for command line strings and for other resources. Now ObjectARX supports one Global and one Local command name, usually one English and one non-English, and can load resources for different languages (if manually implemented). Finally Lisp don’t have a build-in localization feature.</p>
<p>The Autoloader mechanism (.bundle folder and PackageContents.xml) can be used when the application is multi-language (i.e. supporting more than 1 language). The XML support localization for almost all properties and modules, by loading the correct file according to the product language. If you application have only 1 language, even non-English, then there is no need for this localization features (as it will always load the same modules).</p>
<p><strong>Approaches</strong></p>
<p>There are some approaches to create an application that can work on different languages. In some cases, the application will have only one non-English language, for instance, if is designed for a specific market. As already mentioned, if is only 1 language, then no additional work is necessary (as the same modules will load every time).</p>
<p>If your code automatically identify the language and change its command name and interface, <a href="http://through-the-interface.typepad.com/through_the_interface/2009/06/registering-autocad-commands-with-localized-names-using-net.html" target="_self">e.g. using .NET Resources</a>, then great. This post will show the simples approach: separated modules for each language.</p>
<p><strong>Using the PackageContents</strong></p>
<p>Let say an application have 2 languages, English and Portuguese. The idea is to enable the ‘HelloWorld’ command on English systems and ‘OlaMundo’ on Portuguese systems (either by OS, AutoCAD or Regional Settings configured to this language). The images below show it running, of English and Portuguese environments, respectively.</p>
<p><a href="http://adndevblog.typepad.com/.a/6a0167607c2431970b017eeac554b3970d-pi"><img alt="enu" border="0" height="102" src="/assets/image_967220.jpg" style="background-image: none; padding-left: 0px; padding-right: 0px; display: inline; padding-top: 0px; border-width: 0px;" title="enu" width="378" /></a></p>
<p><a href="http://adndevblog.typepad.com/.a/6a0167607c2431970b01901bc7ed5c970b-pi"><img alt="ptb" border="0" height="111" src="/assets/image_814984.jpg" style="background-image: none; padding-left: 0px; padding-right: 0px; display: inline; padding-top: 0px; border-width: 0px;" title="ptb" width="420" /></a></p>
<p>For this case, there are 2 DLLs, that register and have all resources for both languages. The goal is to ensure only the correct DLL will be loaded on each system. This approach can be used for DLLs, ARX, DBX, Lisp or CUIx files, just specify which file is loaded on each supported locale and the Autoloader will make sure the correct version is loaded.</p>
<p>The following XML code show some attributes suffixed with <strong>Ptb</strong>, which then contain the specific version of the data, and the affected lines are marked in <span style="background-color: #ffff00;">yellow</span>. Note the <strong>OnlineDocumentation</strong> and <strong>ModuleName</strong> attributes. The last marked lines contains the commands names, <strong>Global</strong> and <strong>Local</strong>, where the Local is suffixed by <strong>Ptb</strong>. Other markup, line <strong>Description</strong>, are required so the user can understand what the application will do. The <strong>HelpFile</strong> will make sure the proper help is opened when the user press F1.</p>
