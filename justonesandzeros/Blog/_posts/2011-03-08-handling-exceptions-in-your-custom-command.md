<p><img alt="" src="/assets/TipsAndTricks2.png" /></p>
<p>If you have a custom command and an exception is thrown within your code, good programming practice is to handle that exception and not let it bubble up to the hosting application.&#0160; The reason is that the hosting application may not know how to handle the exception properly.&#0160; Some API&#39;s may have built in support for passing specific types of exceptions up the stack, but Vault Explorer does not have one of those APIs.</p>
<p>Let me explain what happens when you throw an exception from a custom command within Vault Explorer.&#0160; If the exception is directly on Vault Explorer&#39;s call stack, it will get &quot;eaten&quot; by Vault Explorer.&#0160; In other words, Vault Explorer will catch the exception then do nothing with it. This is the intended behavior for Vault Explorer.&#0160; However I have recently become aware of a few cases that result in un-intended behavior.</p>
<p>One interesting back door is when the exception comes from an event in custom UI.&#0160; For example, your command launches a custom dialog, and your OnClick event code has an exception that you don&#39;t handle.&#0160; In this case, the Vault Explorer treats it like an exception from any of the built-in dialogs and attempts to handle the issue.&#0160; Usually the handling involves parsing the error and displaying a formatted error dialog.&#0160;</p>
<p>In the case of server errors (SOAPExceptions), this is great because Vault Explorer will parse the error for you and display it in the proper language.&#0160; <br /><img alt="" src="/assets/SOAPError.png" /> <br />I&#39;ll admit, this is pretty nice back door.&#0160; The only downside that I can see is that the error text was written with Vault Explorer in mind and may not make sense in the context of your application.</p>
<p>But if the exception is not a SOAP Exception you will likely get a CER dialog.&#0160; If you don&#39;t know what a CER dialog is, let me show you a picture.   <br />&#0160;<img alt="" src="/assets/CER.png" /> <br />Recognize it now?&#0160; You want to avoid this dialog.</p>
<p>An even worse case comes up if your command spawns a thread which bubbles up an exception.&#0160; In that case, the CER pops up and the application crashes.   <br /><img alt="" src="/assets/AppCrash.png" />&#0160; <br />You <strong>really</strong> want to avoid this dialog.</p>
<p>&#0160;</p>
<p>The best way to avoid these bad cases is to handle your exceptions.&#0160; When in doubt, you can always wrap everything in a try/catch block.&#0160; See examples below.</p>
