<p>One interesting little API gem is the ExtensionLoader.&#0160; Well, it’s interesting to me because I designed it, but maybe you will find it interesting too.&#0160; I basically took the mechanism for loading plug-ins and made it public in the API.&#0160; In theory, you can make use of this tool to load extensions of your own.&#0160; Another use is interrogating the system to see which Vault plug-ins are available, where they live, what their settings are, and so on.</p>
<p>In order to be loaded by the framework, a plug-in has to meet certain requirements.</p>
<ul>
<li>The plug-in has to work through an interface.</li>
<li>The plug-in needs a class that implements the interface.</li>
<li>The loader app needs to be aware of the interface.</li>
<li>The implementing class needs a default constructor.</li>
<li>There needs to be .vcet.config file, which is what tells the loader framework about the plug-in.</li>
</ul>
<p>If you take a closer look at the .vcet.config for a 2014 plug in, you will notice that an &lt;extension&gt; contains an <strong>interface</strong> and <strong>type</strong>.&#0160; <strong>Interface</strong> can be anything you want.&#0160; IExplorerExtension, IJobHandler and IWebServiceExtension are the ones provided by the Vault API.&#0160; But the loader framework doesn’t care about that.&#0160; It will work with any .NET interface.&#0160; <strong>Type</strong> is the class that implements the interface.&#0160; The Loader creates a new instance of that class when loading the plug-in.</p>
<hr noshade="noshade" style="color: #d09219;" />
